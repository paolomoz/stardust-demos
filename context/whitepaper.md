# Whitepaper variants — for narration and framing

Three proposal variants exist as samples on the Stardust site under `samples/stardust-whitepaper/`. Skim them when shaping demo voiceover and slide framing — they carry the most polished phrasing.

| Variant | Title / framing | Best for |
|---|---|---|
| proposed-A | "Replatforming, _reimagined._" — vision brief, exec-flavored | Opening the AEM combo demo / setup narration |
| proposed-B | Same banner; sharpens the **GTM strategy** ("ship the demo before the deck") | Sales-internal framing of the demos themselves |
| proposed-C | Full pipeline story (Stardust → Snowflake → EDS), most operational detail | Ground truth for the AEM-context demo script |

## Pull-quotes worth memorizing

From A and B:

- "A platform migration sold on Core Web Vitals never reaches the exec table. A brand-presence refresh does."
- "The cost of uniqueness collapsed."
- "Every page can be its own page again — provided the brand keeps the rules."
- "Apple-grade craft, on every page."
- "Tokens, not templates."
- "Ship the demo before the deck."
- "One prompt · one website."

From C:

- "Stardust outputs clean, semantic markup. Snowflake maps it onto Edge Delivery's blocks without a hand-off step."
- "Reuse everything. Build nothing new" → "Every page can be its own page again."

## Strategic insight to preserve

The wedge isn't the platform migration; it's the **design fatigue** that turns a technical migration into a board-level priority. Demos should lead with the visual outcome, then reveal the pipeline.
